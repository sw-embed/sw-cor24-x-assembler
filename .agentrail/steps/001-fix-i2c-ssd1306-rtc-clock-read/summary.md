Fixed i2c_ssd1306_rtc_clock.s read-runaway bug. Root cause: bus state machine's slave_sda_pull flips to true after the master-ACK/NAK clock's SCL fall (phase = TxByte{0,0}, tx_byte = 0 from the byte's bit shifts → slave_pull = MSB == 0 = true). Open-drain wired-AND then suppresses i2cstop's SDA-up and STOP never fires; bus stays in transaction; slave keeps streaming bytes through register file and into RAM. Fix: skip the explicit 9th clock entirely on the last read — let i2cstop's SCL-up act as the ACK clock. At that moment slave_pull is still false (from prior AckSlaveToMaster fall, line 310 catch-all), so the subsequent SDA-up fires STOP cleanly. Side effect: spurious on_master_ack() + pointer advance on DS1307, harmless because next iteration resets pointer. Implementation: new i2cread_last subroutine (8-bit read loop with no trailing clock); 3rd read in main_loop uses it. Verified: exactly 3 RDs + STOP per iteration, OLED renders 12:34:56 against ds1307@0x68?hour=12&minute=34&second=56. No regressions in i2c_ds1307_read, i2c_ssd1306_hello, i2c_add1_ping. All workspace tests + clippy clean.
